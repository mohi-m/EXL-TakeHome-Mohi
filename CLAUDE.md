# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Hard Constraints

These rules must never be violated:

- **Local-only.** No cloud services, no paid APIs, no network calls of any kind.
- **SQLite only.** Use Python's stdlib `sqlite3` module. The database lives at `outputs/curated.sqlite`.
- **Never hard-code business-answer values.** All outputs in `outputs/business_answers.md` must be generated by querying `input_data/` through the pipeline.
- **Curated schema is derived from `input_data/sttm_target_mapping.csv`.** Do not define target columns or transformation rules anywhere else.
- **DQ checks are derived from `input_data/data_quality_rules.csv`.** Do not hard-code rule logic that contradicts or bypasses that file.
- **No silent drops.** Every record that fails a FK check or value validation must land in the `dq_exception_report` table (and `outputs/exceptions.csv`), not be discarded.
- **Risky columns load as TEXT in staging.** Timestamps (e.g. `order_ts`, `created_ts`) and monetary amounts (e.g. `order_total`, `amount`) are loaded as TEXT in the raw staging tables; type-casting and parsing happen in `src/transform.py`.

## Coding Standards

- **Docstrings on every function.** Every `def` must have a docstring explaining what it does, why any non-obvious decisions were made, and what it returns. One-liners are fine for trivial helpers; multi-line for anything with logic.
- **Inline comments on complex code.** Any loop, expression, or SQL fragment that isn't immediately obvious to a reader must have a short comment explaining the *why*, not the *what*.
- **Consistent logging.** Every Python module must obtain its logger with `logger = logging.getLogger(__name__)` at module level. Use `logging.basicConfig(...)` only in `if __name__ == "__main__"` blocks, never inside functions or at import time. Log levels: `INFO` for stage boundaries and row counts, `DEBUG` for schema/column details, `WARNING` for unexpected-but-recoverable conditions, `ERROR` for failures before raising.

## Architecture

This is a four-stage local data pipeline for OmniRetail, storing everything in a single SQLite database (`outputs/curated.sqlite`) with no external infrastructure required.

**Stage flow** (`src/pipeline.py` orchestrates):
1. `src/ingest.py` — Loads raw source files into SQLite staging tables as-is: `customers.csv`, `products.csv`, `orders.csv`, `payments.csv`, `support_tickets.jsonl`
2. `src/transform.py` — Applies STTM rules (`input_data/sttm_target_mapping.csv`) to produce five curated tables: `dim_customer`, `dim_product`, `fact_order`, `fact_payment`, `fact_customer_issue`. Rows with invalid FK references are routed to an exceptions table rather than dropped.
3. `src/quality_checks.py` — Executes DQ001–DQ012 rules (defined in `input_data/data_quality_rules.csv`) against the curated tables and writes failures to `outputs/exceptions.csv`
4. `src/reporting.py` — Runs the five business-question queries and renders `outputs/data_quality_report.md` and `outputs/business_answers.md`

**Key data transformations** (from `input_data/sttm_target_mapping.csv`):
- Mixed timestamp formats in `orders.csv` → normalized to date via `strptime` fallback
- Duplicate customers resolved before FK linking so all downstream tables reference a single canonical `customer_key`
- Country normalization: "USA", "US", "United States" → "USA"
- State names mapped to two-letter codes where possible

**SQL files** (reference/companion to Python logic):
- `sql/curated_model.sql` — DDL and transformation logic for the five curated tables
- `sql/business_questions.sql` — The five analytics queries (revenue by month, top customers, order exceptions, revenue by state, support ticket correlation)

**Tests** (`tests/test_quality_checks.py`) validate DQ001–DQ012 rules against the curated tables via pytest.

## Data Quality Rules

Rules are driven by `input_data/data_quality_rules.csv`. High-severity rules (DQ001, DQ004–DQ010) enforce uniqueness and FK integrity. Medium-severity rules (DQ002–DQ003, DQ011–DQ012) cover format/standardization checks. Exception records are preserved in the output, not silently dropped.

## Output Files

| File | Description |
|---|---|
| `outputs/curated.sqlite` | Curated SQLite DB with all five tables |
| `outputs/data_quality_report.md` | DQ rule results (DQ001–DQ012) |
| `outputs/exceptions.csv` | Records that failed FK or value checks |
| `outputs/business_answers.md` | Answers to the five business questions |
